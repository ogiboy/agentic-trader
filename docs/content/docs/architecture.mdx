## Staged runtime shape

The current runtime is a staged specialist graph:

1. Research Coordinator
2. Fundamental Analyst
3. Macro / News Analyst
4. Regime Agent
5. Strategy Selector
6. Risk Agent
7. Specialist Consensus Layer
8. Manager Agent
9. Execution Guard
10. Paper Broker

## Package ownership

- `agentic_trader/agents/`: specialist, manager, and operator-facing reasoning surfaces.
- `agentic_trader/llm/`: provider access and role-based model routing.
- `agentic_trader/market/`: data loading, market context, calendar, and feature preparation.
- `agentic_trader/providers/`: provider boundaries and canonical aggregation.
- `agentic_trader/memory/`: retrieval, embeddings, and memory policy.
- `agentic_trader/storage/`: DuckDB persistence.
- `agentic_trader/engine/` and `agentic_trader/execution/`: guard logic, adapters, execution intent/outcome, and broker seams.
- `agentic_trader/workflows/`: one-shot, service, replay, and backtest workflows.

## Operator-surface rule

CLI, Rich, Ink, observer API, and Web GUI should consume the same runtime truth.
Do not solve a backend inconsistency by adding UI-only logic.

## Frontend rule

The repo direction is:

- Next.js App Router
- Tailwind v4 token themes
- shadcn source components with the current `radix-lyra` + `olive` setup
- thin route handlers that delegate to existing runtime contracts

The Web GUI remains a thin local shell.
The docs app is a developer-facing shell.
Neither should become a second runtime.
