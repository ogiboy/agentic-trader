## Agentic Trader bugün ne yapabilir

Agentic Trader artık yalnızca bir prototip değil.
Bugün şunları yapabilir:

- mevcut Python runtime üzerinden strict lokal paper cycle çalıştırabilir
- market context, decision feature, review snapshot ve runtime artefact'lerini
  oluşturup persist edebilir
- runtime, provider, broker, research ve supervisor durumunu CLI, Rich, Ink,
  observer API ve Web GUI yüzeylerinde gösterebilir
- lokal paper backend'i varsayılan tutarken live execution'ı bloklu tutabilir
- model/provider sağlığı, runtime mode ve Alpaca paper hazırlığı için readiness
  gate'lerini gösterebilir
- smoke koşuları ve evidence bundle'lar ile QA kanıtı paketleyebilir

Docs bu ürün gerçeğini sade operatör diliyle açıklamalı, implementation
detaylarına ise gerçekten yardımcı oldukları yerde bağlanmalı.

## Hâlâ sertleştirilen alanlar

Mevcut hardening hattı şunlara odaklanıyor:

- runtime, provider ve model sağlığını daha görünür yapmak
- onboarding ve kurulum kalitesini yükseltmek
- web yüzeylerini ince ve contract-aligned tutmak
- docs'u yalnızca katkı verenlere değil operatörlere de faydalı hale getirmek
- research-sidecar evidence ingestion'ı broker veya execution yetkisi vermeden
  genişletmek

## Hazır olmayan veya V1 kapsamı dışında kalanlar

- Live trading hâlâ bloklu.
- IBKR, global marketler, FX ve multi-currency accounting V2 kapsamı.
- CrewAI research flow izole sidecar'dır; ana runtime değildir.
- SEC EDGAR submissions metadata opt-in ve metadata-first çalışır; full filing
  text, transcript, macro series, KAP ve daha zengin vendor ingestion hâlâ
  provider-level gelecek iştir.
- Web GUI ve docs runtime contract'lerinin etrafındaki lokal kabuklardır;
  alternatif trading engine değildir.

## Drift etmemesi gereken şeyler

- Kaynak gerçek hâlâ Python runtime'dır.
- `webgui/` mevcut runtime sözleşmelerine delegasyon yapar; onların yerine geçmez.
- `docs/` ürün davranışını ve katkı sınırlarını açıklar; runtime değildir.
- Varsayılan execution posture paper trading olarak kalır.
- Fallback yolları açık ve inspect edilebilir olmalıdır.

## Ürün tarafında açık girişimler

### İsteğe bağlı app-managed Ollama supervision

Hedef davranış, Ollama'yı runtime diagnostiklerinde zaten kullanılan aynı status ve log yüzeyi üzerinden yönetebilmek.

Bunun anlamı:

- açık start ve stop davranışı
- görünür health check'ler
- okunabilir request ve log kanıtı
- model takıldığında sahte “healthy” durumu olmaması

### Cross-platform bootstrap

Daha uzun vadeli onboarding hedefi; macOS, Linux ve Windows için şu işleri yapabilen yönlendirmeli bir kurulum akışı:

- prerequisite denetimi
- isteğe bağlı bağımlılık kurulum teklifleri
- Python ve Node ortam hazırlığı
- provider ve model ready oluş doğrulaması
- doğru lokal yüzeyin başlatılması

## Docs açısından mevcut duruş

Bu docs sitesi artık yalnızca kısa bir kurulum metni olmamalı.
Şunları da açıklamalı:

- mevcut ürün kabiliyetleri
- paper/live/runtime davranışlarının arkasındaki guardrail'ler
- bir paper run'ın nasıl doğrulanacağı
- memory, review artefact'leri, broker durumu ve evidence bundle'ların güveni
  nasıl desteklediği
- bir özelliğin tamam mı, gated mı, scaffold mı, yoksa gelecek kapsam mı olduğu
